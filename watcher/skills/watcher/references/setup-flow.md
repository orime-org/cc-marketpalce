# Setup Flow — configure 模式建/改 `.watcher/` 流程

`/watcher configure` 时执行本流程，建或改 `.watcher/` 项目级配置。

## 触发条件

| 用户输入 | 行为 |
|---|---|
| `/watcher configure` 或用户明示"配置 watcher" | **执行本流程** |
| Stop hook 自动触发的 watcher audit | **绝不执行** ——audit 摘要里提示用户手动跑即可 |

## 状态分支

| `.watcher/` 状态 | 模式 |
|---|---|
| 不存在 | 首次建 |
| 存在但缺 1-2 个文件 | 局部补全（只问缺的那个） |
| 完整但用户想改 | 修订模式 |

## 流程（首次建模式）

### 第一步：报告盘点结果给用户

精简列盘点看到的关键信号：

```
检测到的项目特征：
- 顶层文件：<list>
- docs/ 结构：<list 或 "不存在">
- 关键文件 / 目录：<list>
- 记忆里关于该项目的描述：<引用具体文字（如有）>
```

### 第二步：问用户 4 个问题

```
我需要帮你建 .watcher/ 项目级配置。请回答：

1. 这个项目的**核心目的**？（一两句话）
2. 主要**产出**是什么？（代码 / 文档 / 设计 / 实验数据 / 配置 / 其他）
3. 你希望 watcher 同步**哪些文档**？（列你认为该项目应该有的 *.md 路径 + 用途）
4. 有什么**特殊关注点**？（watcher 每次跑都要检查的东西，如"提醒检查 .env 不要进 git"）
```

### 第三步：起草 3 个文件让用户确认

**不要立刻 Write 到磁盘**——先把内容呈现给用户审。

#### `.watcher/project-summary.md` 草稿

```markdown
# 项目目的
<用户第 1 题回答>

# 主要产出
<用户第 2 题回答>

# 项目阶段
vibe / alpha / beta / production
（让用户选，或 watcher 根据盘点结果推断后让用户确认）

# 关键约束 / 红线
<用户主动提的，如"不入 git" / "不用第三方依赖">
```

#### `.watcher/doc-inventory.md` 草稿

```markdown
# 应有文档清单

| 文档路径 | 用途 | 什么时候改 |
|---|---|---|
| README.md | 项目入口 + 怎么跑起来 | 安装步骤变 / 主功能变 |
| docs/X.md | <用户列的> | <根据用途推 hint> |
| ... | ... | ... |

# 缺失文档
（盘点时找不到但应该有的，列在这里供用户决定要不要建）
```

#### `.watcher/watchlist.md` 草稿

```markdown
# 用户自定义关注点

watcher 每次跑都纳入考虑：

- <用户第 4 题列的关注>
- （随时加 / 改 / 删，无需 watcher 介入）
```

### 第四步：用户确认后 Write

等用户回 "OK" / 改 X / 加 Y / 删 Z 后再 Write 到 `<project-root>/.watcher/`：

```
<project>/.watcher/
├── project-summary.md
├── doc-inventory.md
└── watchlist.md
```

**禁止**未经用户确认直接 Write 文件。

### 第五步：提示用户验证

落盘后只输出：

```
已写入 .watcher/，请手动输入 /watcher 验证配置生效后的行为符合预期。
```

**不自动跑 audit**——让用户亲眼看到 watcher 用新配置跑出来啥样。

## 局部补全模式（`.watcher/` 存在但缺某个文件）

按缺哪个补哪个：

| 缺失 | 问什么 |
|---|---|
| project-summary | 问"项目目的 / 产出 / 阶段" |
| doc-inventory | 问"应有文档清单"|
| watchlist | 问"特殊关注点（可空，未来随时加）" |

跟首次建模式相同的"草稿展示 + 用户确认才 Write"约束。

## 修订模式（`.watcher/` 完整，用户想改）

1. 读现有 3 文件 + 项目画像 → 展示
2. 开放问"哪里要改？"（不固定 Q 数量）
3. 用户答 → 改对应草稿（不是重新生成全部）
4. 草稿展示 → 用户 OK 才 Write
5. 同首次建模式第五步

## 用户体验原则

- ✅ 列盘点结果具体（让用户能判断）
- ✅ 问题精简（4 个最多）
- ✅ 起草让用户先看
- ❌ 不要默默写入磁盘
- ❌ 不要问技术细节（文件格式由 watcher 起草，用户只需确认）
- ❌ 不要自动跑 audit（落盘后让用户主动跑）

## 写入位置

所有交互结果写入**项目本地** `<project-root>/.watcher/`：

- ✅ 项目本地 — 跟项目代码同寿命，进 git（如果项目有 git，建议团队共享配置）
- ❌ 不写入 agent 记忆 — 记忆是跨会话个人偏好，项目配置是项目特定，不混淆

## 用户跳过交互的 fallback

用户消息明确说 "你看着办" / "不重要" / "跳过"：

- 提示用户："已记下，将跳过本次配置。watcher 仍可用通用规则跑 audit。"
- 不创建任何 `.watcher/` 文件
- audit 摘要里继续提示"未建 `.watcher/`"
